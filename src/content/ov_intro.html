How Overrides
Work

ACM Extended does not sit beside ACM and ACE. It replaces specific functions inside them. This page is the mechanism: how a replacement is installed, why it has to be done this way, and what that costs you.

The rule we follow

Overrides are additive wherever possible. Where a gate exists, ACM Extended can make it harder to pass but never easier, so nothing here can accidentally save a casualty ACM would have let die.

01

Final Functions & Why Wrapping Fails

The obvious way to change a function's behaviour is to keep a reference to the original, install your own, and call through. That does not work here, and understanding why explains the shape of everything else in this section.

ACM compiles most of its own functions final. A final function cannot be reassigned at runtime. Assigning over it does not raise an error you will notice in normal play; the assignment is simply ignored and the original keeps running. Code that looks installed sits there doing nothing.

The silent failureCosts hours

This is the single most expensive trap in the addon and it has been hit more than once.

A runtime reassignment of a final function produces an Attempt to override final function line in the RPT log and then carries on. If you are not reading the log, the symptom is that your change appears to do nothing at all. Not intermittently. Not partially. Nothing.

The functions this has caught include the EKG generators, ACM_core_fnc_getUp, and ACM_circulation_fnc_getIVFlowRate. In each case the code was correct and simply never ran.

How to check. Search the RPT for final function. Anything named there is a function you cannot reassign at runtime and must replace at compile time instead.

Compile-time replacementCfgFunctions

The mechanism that does work is declaring the function in CfgFunctions under the same name, from an addon that loads after the one you are replacing. Load order decides the winner, and ACM Extended declares ACM and ACE as dependencies, so it always loads later.

The override classes live in config.cpp and are named by the subsystem they touch:

02

De-Macroing & Re-Pull Discipline

Compile-time replacement means the whole function is yours. There is no original underneath to fall back to, so every line of the original behaviour has to be reproduced in your copy or it is gone.

That has a maintenance consequence you should understand before extending anything here. When ACM or ACE change a function we have replaced, our copy does not inherit the change. It keeps running the old logic against a newer game. Nothing warns you.

De-macroingWhy the files are long

ACM and ACE source is written with preprocessor macros: GET_OXYGEN, GET_IV, QEGVAR and so on. Those macros are defined in the originating addon's header and resolve against that addon's tag.

A copy compiled inside ACM Extended resolves those macros in our tag context, which either fails to compile or silently reads the wrong variable. So every replacement is de-macroed by hand: the macro is expanded to the literal variable name it produces.

That is why fn_getIVFlowRate.sqf writes ACM_circulation_IV_Placement in full rather than GET_IV, and why these files are longer than the originals they replace. It is deliberate, and it is the only way the copy behaves identically.

Re-pull disciplineOn every ACM update

Each override file carries a comment naming what it replaced and what was added. When ACM or ACE update, the procedure is:

  1. Diff the upstream function against the version our copy was taken from.
  2. If upstream changed, re-pull the new original, de-macro it again, and re-apply our additions on top.
  3. Never patch our copy to match a described change. Take the new source.

The additions are deliberately small and clearly marked in each file precisely so this step stays cheap. A replacement that has drifted into a rewrite cannot be re-pulled and becomes permanent maintenance.
