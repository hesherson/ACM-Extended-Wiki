PILLAR // ADVANCED FUNCTIONS

Known
Traps

Failures that cost hours to find, collected so they cost minutes instead. Every one of these has actually happened during development, and most produce no error message at all.

The common thread

Every trap here fails silently. Nothing throws, nothing logs where you are looking, and the code appears correct on inspection. That is what makes them expensive.

01

The Trailing Comma

A trailing comma inside an array kills fn_postInit at parse time. Everything registered after that point never registers.

The symptom is not an error. It is that some arbitrary set of features are simply absent, with no pattern that makes sense, because the cut-off is wherever the comma happens to sit in the file rather than anywhere meaningful.

Looks likeActually is
Half the actions missingParse stopped partway down postInit
A feature that worked yesterdaySomething above it gained a comma
No RPT errorCorrect. There is none.

It is checked on every build by commacheck.py, which is the only reliable defence. Do not rely on spotting it by eye; the comma is invisible in a long array.

02

Action Condition Inheritance

An action that inherits from a hidden parent inherits the parent's condition, including a condition of false. The child then never appears, and the child's own definition looks completely correct because the problem is not in it.

Restate conditions explicitly on every child. Do not rely on inheritance to leave them alone.

The thoracostomy caseSame family

A closely related version of this took a long time to find. The thoracostomy action was not appearing, and its condition was correct.

The override had restated condition but silently inherited ACM's items[] = {"ACE_surgicalKit"}. ACE's medical menu filters any action whose required items the medic is not carrying, so the action vanished whenever the surgical kit was absent, which looked exactly like a broken condition.

The fix was overriding items[] = {} as well. The general form: when replacing an action, check every inherited property, not only the one you meant to change.

03

Final Functions

ACM compiles most of its functions final. A runtime reassignment is ignored, the original keeps running, and your code never executes. The full mechanism is on the How overrides work page; this is the checklist version.

SymptomCheck
Change has no effect whatsoeverRPT for final function
Feature appears entirely absentA gate upstream returning false
Value oscillates between two figuresTwo writers on a cycled variable

Functions confirmed final and therefore requiring compile-time replacement include the EKG generators, getUp, getIVFlowRate, getBloodPressure and updateHeartRate. Functions confirmed not final include updatePeripheralResistance and getBloodVolumeChange.

04

Two Writers, One Value

Where a variable is recomputed on a cycle, writing it from your own tick does not give you ownership. Your write and the cycle's write alternate, and the value visibly flickers.

This has caused three separate bugs: the ventilator respiratory rate against ACM's vitals cycle, saturation against ACE's gas exchange, and the altitude model against the ventilator shunt model.

ValueRule
Respiratory rateOwn updateRespirationRate, do not write the variable
SaturationWrite the ACM target layer only, never ace_medical_spo2
Saturation while ventilatingThe vent owns it. Every other writer carries the guard.
Heart rateWrite both variables, always

Before Shipping

  • Run the checks. Brace balance, duplicate config classes with the button macro expanded, and trailing commas. All three catch silent failures.
  • Check for duplicate CAManBase entries. ACE_Actions and ACE_SelfActions must appear exactly once each.
  • Sweep for use before define. A function called before its definition fails only on the path that reaches it.
